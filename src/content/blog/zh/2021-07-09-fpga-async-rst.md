我们经常讨论同步复位和异步复位。不过读者需要注意，“异步复位”这个说法可能指的是下面几件不同的事情：

1. 寄存器使用的是异步复位引脚，而不是同步复位引脚。
2. 复位信号相对于你的时钟是异步到达的。
3. 你的整体复位设计策略。

下面分别来看。

## 带异步复位的寄存器

先看一段 Verilog 代码：

```verilog
  always @ (posedge clk or posedge arst) begin
    if (arst) begin
      data <= 1'b0;
    end else if (cen) begin
      data <= din;
    end
  end
```

这段代码本身没有什么特别之处。`arst` 拉高时，`data` 会被复位为 `1'b0`；`arst` 释放后，又恢复正常工作。由于复位优先级高于时钟，因此即使没有时钟边沿，寄存器的 `Q` 输出也会被清零。综合之后，你会得到一个带异步复位的触发器。

异步复位何时拉高通常并不重要，因为复位期间寄存器输出是有保证的。但如果异步复位在靠近时钟边沿的时候释放，寄存器就可能进入亚稳态。亚稳态意味着输出不可预测，并且存在设计失效的风险。所以，**即使是异步复位，也有类似同步复位的时序要求**。这是很多初学者容易忽略的地方。这里的时序关系叫做 **recovery 和 removal time** 检查。

![Recovery and Removal Time Checks](/assets/images/posts/fpga-async-rst/fpga-async-timing.drawio.png)

**Recovery check** 要求复位信号在有效时钟边沿到来之前至少提前 `recovery time` 稳定下来，这和同步引脚的 `setup time` 很像。**Removal check** 要求复位信号只能在有效时钟边沿之后至少经过 `removal time` 才允许变化，这和 `hold time` 类似。需要注意的是，**recovery/removal 时间只针对异步复位的释放过程**。因为异步复位的拉高会直接控制输出，不依赖时钟边沿。

## 复位信号是异步到达的

异步复位引脚自身的行为并没有问题，只要它满足 recovery/removal 的时序要求即可。真正的问题在于这些时序要求很容易被破坏。比如复位信号来自外部引脚时，没人能保证它具体在什么时候到达。再比如把 PLL 的 lock 指示信号直接拿来当异步复位。来自其他时钟域的复位也应该被视为异步信号。前面已经提到，即使是异步复位，也必须满足 recovery/removal 要求，否则设计就可能出错。

为了避免异步复位信号带来的问题，我们通常会加入“异步复位同步器”电路。这里给出两种最常见的做法。

### 1. 异步复位，同步释放

![Asynchronous Reset, Synchronous Release](/assets/images/posts/fpga-async-rst/fpga-async-rst-async.drawio.png)

几个寄存器串联在一起，这些寄存器的异步复位引脚都接到外部复位输入上。当 `ARST_IN` 拉高时，所有寄存器以及 `ARST_OUT` 都会立刻变成 1。当 `ARST_IN` 释放后，需要经过若干个时钟周期，`ARST_OUT` 才会回到 0。为了滤除异步复位到达时刻不确定带来的亚稳态，这条链路至少需要 2 级触发器。级数越多，平均失效间隔时间（MTBF）通常越高。

它能解决异步复位的所有问题吗？来看下面这个电路：

![Asynchronous Reset Issue](/assets/images/posts/fpga-async-rst/fpga-async-issue.drawio.png)

寄存器 A 的输出送到寄存器 B，它们工作在同一个时钟下，因此寄存器 B 期望接收到的是同步数据。注意，`ARST` 到寄存器 B 的 `D` 引脚实际上构成了一条有效数据路径：`ARST` -> `PRE` -> `Q` -> 寄存器 A 的 `PRE` -> 寄存器 A 的 `Q` -> 寄存器 B 的 `D`。也就是说，复位源会“穿过”一条很长的路径，最终跑到一个同步数据引脚上。由于复位源本身是异步工作的，让一个期望同步输入的触发器去采样这条路径，本质上仍然是设计违规。设计 FSM 等关键控制逻辑时，这一点尤其要小心。

### 2. 完全同步器

另一种同步器如下图所示：

![Synchronous Reset, Synchronous Release](/assets/images/posts/fpga-async-rst/fpga-async-rst-sync.drawio.png)

一个通用的单比特 CDC 电路通常就能满足需求。`RST_OUT` 的拉高和释放都会和 `CLK` 同步，因此它解决了异步复位“穿透”到同步数据路径上的问题。同步后的复位既可以安全地接到寄存器的异步复位引脚，也可以接到同步复位引脚。和方案 1 相比，它的缺点是没有时钟时就无法工作。

|                 | Async Reset | Sync Reset |
|-----------------|-------------|------------|
| 同步器 #1       | OK          | NOK        |
| 同步器 #2       | OK          | OK         |

## 同步复位与异步复位

如果你可以自己决定寄存器使用哪种复位类型，也不用过度焦虑。同步复位和异步复位都能有效实现设计所需的复位功能，只是各自有优缺点。如果你需要在时钟启动之前，甚至在没有时钟的情况下，就把某一部分逻辑复位，那么同步复位就无能为力。如果复位信号本身又是异步到达的，那么方案 1 仍然是唯一选择。

ASIC 设计中通常更偏向使用异步复位，因为这样可以节省晶体管数量。而 Xilinx 则更推荐在 FPGA 里使用同步复位。这可能是因为他们的大多数原语本身就已经很好地支持同步复位，综合工具也能更充分地利用同步复位的可替换性来优化设计。

最后给一个简单总结，说明不同复位类型和同步器的选择关系：

|             | Gated Clock     | No Gated Clock  |
|-------------|-----------------|-----------------|
| 异步复位    | 同步器 #1       | 两者都可以      |
| 同步复位    | 不可行          | 同步器 #2       |
